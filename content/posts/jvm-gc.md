# JVM 垃圾回收看这篇就够了

JVM 垃圾回收是面试八股文里的重头戏，也是我觉得最难啃的一块。概念多、算法多、收集器也多，第一次看的时候感觉脑子不够用。这篇把我学到的整理一下，争取讲清楚。

![垃圾回收示意图](https://upload.wikimedia.org/wikipedia/commons/thumb/3/3c/Tracing-garbage-collection.svg/400px-Tracing-garbage-collection.svg.png)

## 堆内存长什么样

JVM 的堆分成两大块：新生代（Young Generation）和老年代（Old Generation）。

新生代又分三块：Eden 区和两个 Survivor 区（S0、S1），默认比例 8:1:1。

为什么这么分？研究发现大部分对象都是"朝生夕死"的，活不过一次 GC。放在新生代用快速算法回收，少数活得久的晋升到老年代，减少回收频率。

对象一般在 Eden 区创建。Eden 满了触发 Minor GC，活着的对象搬到 Survivor 区。每熬过一次 GC 年龄加 1，到了阈值（默认 15）就晋升到老年代。

顺便说一下，两个 Survivor 区同一时刻只有一个在用，另一个是空的。GC 的时候把 Eden 和在用的 Survivor 里的存活对象一起复制到空的那个 Survivor，然后清空原来的区域。这就是复制算法在新生代的应用。

## 怎么判断对象该不该回收

主流方案是**可达性分析**。从一组叫 GC Roots 的对象出发，沿引用链往下找，找不到的对象就是垃圾。

哪些东西算 GC Roots？

- 虚拟机栈中引用的对象（方法里的局部变量）
- 类的静态变量引用的对象
- 常量引用的对象
- JNI 引用的对象
- 同步锁持有的对象

你可能听过另一种方案叫引用计数法——每个对象记个计数器，被引用就加 1，引用失效就减 1，计数为 0 就回收。听着很简单对吧？但有个致命问题：循环引用。A 引用 B，B 引用 A，计数永远不为 0，但其实都是垃圾。所以 JVM 没用引用计数。

Python 用的引用计数，但额外做了循环引用检测。每种语言的 GC 思路还是挺不一样的。

## 三种回收算法

**标记-清除（Mark-Sweep）**

分两步：先标记出所有要回收的对象，然后统一清除。

问题是清除之后内存会产生很多碎片，大对象找不到连续空间分配。

**复制算法（Copying）**

把内存分成两半，只用其中一半。GC 的时候把活着的对象复制到另一半，然后把这一半整个清空。

优点是没有碎片，缺点是浪费了一半内存。新生代的 Survivor 区就是这个思路，不过因为大部分对象活不久，实际复制的量很少，所以效率很高。

**标记-整理（Mark-Compact）**

先标记，然后把活着的对象都往一端移动，最后清掉边界以外的内存。

没有碎片，也不浪费空间，但移动对象开销大。老年代一般用这个。
