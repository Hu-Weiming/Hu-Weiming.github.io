# MySQL 索引优化，别再全表扫描了

上学期数据库课的大作业，我写了个选课系统。数据量小的时候跑得飞快，一导入几万条测试数据就卡得不行。一查原因——全表扫描。那时候我才真正意识到索引的重要性。

## B+树——索引的底层结构

MySQL 的 InnoDB 引擎用 B+树来组织索引。为啥是 B+树不是别的？

![B-tree 结构示意图](https://upload.wikimedia.org/wikipedia/commons/thumb/6/65/B-tree-definition.svg/831px-B-tree-definition.svg.png)

B+树有几个特点：
- 非叶子节点只存 key，不存数据，这样一个节点能存更多的 key
- 叶子节点存所有数据，而且用链表串起来
- 所有查询都要走到叶子节点，查询效率稳定

为啥不用二叉搜索树或者红黑树？因为磁盘 IO 是瓶颈。B+树很矮很胖，一个节点通常是一个磁盘页（16KB）。三层的 B+树就能存上千万行数据，意味着查找一条数据最多三次磁盘 IO。

我算过一笔账：假设主键是 bigint（8 字节），指针 6 字节，一个页能存 16KB / 14B ≈ 1170 个 key。叶子节点假设每行 1KB，一个页存 16 行。三层就是 1170 × 1170 × 16 ≈ 两千万行。这也是为啥大家说"两千万以内不用分表"。

## 聚簇索引 vs 非聚簇索引

这俩概念必须搞清楚。

**聚簇索引**（主键索引）：叶子节点直接存整行数据。InnoDB 的数据就是按主键顺序组织的，所以一张表只能有一个聚簇索引。

**非聚簇索引**（二级索引）：叶子节点存的是主键值。通过二级索引查到主键后，还要拿着主键去聚簇索引里查完整数据，这个过程叫**回表**。

```sql
-- 假设 name 上有索引
SELECT * FROM student WHERE name = '张三';
-- 1. 在 name 索引的 B+树里找到"张三"，拿到主键 id=1001
-- 2. 拿着 id=1001 去聚簇索引查完整的行 → 这就是回表
```

回表意味着两次 B+树查找。数据量大的时候这个开销不小。
