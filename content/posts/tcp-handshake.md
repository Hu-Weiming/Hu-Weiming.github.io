# TCP 三次握手和四次挥手，画个图就懂了

## 三次握手流程

TCP 三次握手大概是网络面试被问得最多的知识点了。背归背，但搞懂"为什么"比记住流程更重要。

先看图：

![TCP 三次握手](https://upload.wikimedia.org/wikipedia/commons/thumb/9/98/Tcp-handshake.svg/800px-Tcp-handshake.svg.png)

用大白话描述一下这三次：

**第一次握手**：客户端发一个 SYN 包给服务端，带上自己的初始序列号（ISN），说"哥们，我想跟你建立连接"。这时客户端进入 SYN_SENT 状态。

**第二次握手**：服务端收到后，回一个 SYN+ACK 包，带上自己的初始序列号，同时确认收到了客户端的 SYN。意思是"收到了，我也想跟你建立连接"。服务端进入 SYN_RCVD 状态。

**第三次握手**：客户端收到后，再发一个 ACK 包，确认收到了服务端的 SYN。这时双方都进入 ESTABLISHED 状态，连接建立完成。

```
客户端                    服务端
  |                        |
  |------- SYN(seq=x) ---->|  第一次握手
  |                        |
  |<-- SYN+ACK(seq=y,ack=x+1) --|  第二次握手
  |                        |
  |------- ACK(ack=y+1) -->|  第三次握手
  |                        |
  |     连接建立完成         |
```

## 为什么是三次，不是两次

这是面试最爱追问的。你可以从两个角度回答。

角度一：确认双方的收发能力。三次握手之后，双方都确认了"我能发、我能收、对方能发、对方能收"。如果只有两次，服务端没法确认客户端的接收能力。

角度二（更本质）：防止已失效的连接请求到达服务端。假设客户端发了一个 SYN，因为网络延迟一直没到，客户端超时后重发了一个新的 SYN 并成功建立连接。等通信结束后，那个迟到的 SYN 到了服务端。如果只有两次握手，服务端收到就直接建立连接了，但客户端根本不知道，就白白浪费了服务端资源。有了第三次握手，服务端等客户端的 ACK，客户端不会回 ACK（因为它没发新的 SYN），连接就不会误建。

我当初看书的时候第一个角度好理解，第二个角度想了一会儿才想通。
