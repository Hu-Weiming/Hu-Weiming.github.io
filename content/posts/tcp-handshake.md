# TCP 三次握手和四次挥手，画个图就懂了

## 三次握手流程

TCP 三次握手大概是网络面试被问得最多的知识点了。背归背，但搞懂"为什么"比记住流程更重要。

先上图：

![TCP 三次握手](https://upload.wikimedia.org/wikipedia/commons/thumb/9/98/Tcp-handshake.svg/800px-Tcp-handshake.svg.png)

用大白话描述一下：

**第一次握手**：客户端发 SYN 包，带上初始序列号（ISN），说"我想跟你建立连接"。客户端进入 SYN_SENT 状态。

**第二次握手**：服务端收到后，回 SYN+ACK 包，带上自己的初始序列号，同时确认收到了客户端的 SYN。"收到了，我也想连你"。服务端进入 SYN_RCVD 状态。

**第三次握手**：客户端收到后，再发 ACK 包，确认收到了服务端的 SYN。双方进入 ESTABLISHED 状态，连接建立完成。

```
客户端                    服务端
  |                        |
  |------- SYN(seq=x) ---->|  第一次握手
  |                        |
  |<-- SYN+ACK(seq=y,ack=x+1) --|  第二次握手
  |                        |
  |------- ACK(ack=y+1) -->|  第三次握手
  |                        |
  |     连接建立完成         |
```

## 为什么是三次，不是两次

面试最爱追问的点。可以从两个角度回答。

角度一：确认双方的收发能力。三次握手之后，双方都确认了"我能发、我能收、对方能发、对方能收"。两次的话，服务端没法确认客户端是不是真的能收到。

角度二（更本质）：防止历史连接。假设客户端发了一个 SYN，因为网络延迟一直没到，客户端超时后重发了新的 SYN 并成功建立连接。通信结束后，迟到的旧 SYN 到了服务端。两次握手的话，服务端收到就建立连接了，但客户端压根不知道，白白浪费服务端资源。有了第三次握手，服务端得等客户端的 ACK，客户端不会理这个旧的 SYN-ACK，连接就不会误建。

## 四次挥手流程

断开连接比建立连接多一次，因为 TCP 是全双工的，每个方向的连接需要单独关闭。

```
客户端                    服务端
  |                        |
  |------- FIN(seq=u) ---->|  第一次挥手：客户端说"我发完了"
  |                        |
  |<------ ACK(ack=u+1) ---|  第二次挥手：服务端说"知道了"
  |                        |
  |   (服务端可能还有数据要发)  |
  |                        |
  |<------ FIN(seq=w) -----|  第三次挥手：服务端说"我也发完了"
  |                        |
  |------- ACK(ack=w+1) -->|  第四次挥手：客户端说"收到"
  |                        |
  |   TIME_WAIT (2MSL)     |
  |                        |
```

为什么不能把第二次和第三次合并成三次挥手？因为服务端收到客户端的 FIN 时，可能还有数据没发完。它先回个 ACK 表示"我知道你要关了"，等自己数据发完了，再发 FIN。

当然如果服务端确实没有数据要发了，第二次和第三次可以合并（TCP 的延迟确认机制），实际抓包有时候能看到三次挥手。

## TIME_WAIT 是啥

四次挥手之后，主动关闭方不是立刻进入 CLOSED 状态，而是先进入 TIME_WAIT，等 2 个 MSL（Maximum Segment Lifetime，报文最大存活时间）才真正关闭。

为什么要等？两个原因：

1. **保证最后一个 ACK 能到达**：如果最后的 ACK 丢了，服务端会重发 FIN。客户端在 TIME_WAIT 状态还能收到并重新 ACK。如果客户端已经关了，服务端重发的 FIN 没人理，就卡住了。

2. **让旧连接的报文消亡**：等 2MSL 确保网络中残留的报文都过期了，不会影响新建的连接。

TIME_WAIT 在高并发短连接场景下会是个问题，大量连接卡在 TIME_WAIT 占端口。可以调内核参数 `tcp_tw_reuse` 来优化，但要注意副作用。
