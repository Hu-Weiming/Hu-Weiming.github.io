# TCP 三次握手和四次挥手，画个图就懂了

## 三次握手流程

TCP 三次握手大概是网络面试被问得最多的知识点了。背归背，但搞懂"为什么"比记住流程更重要。

先上图：

![TCP 三次握手](https://upload.wikimedia.org/wikipedia/commons/thumb/9/98/Tcp-handshake.svg/800px-Tcp-handshake.svg.png)

用大白话描述一下：

**第一次握手**：客户端发 SYN 包，带上初始序列号（ISN），说"我想跟你建立连接"。客户端进入 SYN_SENT 状态。

**第二次握手**：服务端收到后，回 SYN+ACK 包，带上自己的初始序列号，同时确认收到了客户端的 SYN。"收到了，我也想连你"。服务端进入 SYN_RCVD 状态。

**第三次握手**：客户端收到后，再发 ACK 包，确认收到了服务端的 SYN。双方进入 ESTABLISHED 状态，连接建立完成。

```
客户端                    服务端
  |                        |
  |------- SYN(seq=x) ---->|  第一次握手
  |                        |
  |<-- SYN+ACK(seq=y,ack=x+1) --|  第二次握手
  |                        |
  |------- ACK(ack=y+1) -->|  第三次握手
  |                        |
  |     连接建立完成         |
```

## 为什么是三次，不是两次

面试最爱追问的点。

角度一：确认双方收发能力。三次握手后，双方都确认了"我能发、我能收、对方能发、对方能收"。两次的话，服务端没法确认客户端是不是真的能收到。

角度二（更本质）：防止历史连接。假设客户端发了一个 SYN，因为网络延迟一直没到，客户端超时重发了新的 SYN 并完成连接。通信结束后，迟到的旧 SYN 才到服务端。两次握手的话，服务端收到就建连了，但客户端压根不知道，白白浪费资源。有第三次握手，服务端得等 ACK，客户端不会理旧的 SYN-ACK，连接就不会误建。

我当初看书的时候第一个角度好理解，第二个角度想了一会儿才明白。

## 四次挥手流程

断连比建连多一次，因为 TCP 是全双工的，每个方向的连接要单独关。

```
客户端                    服务端
  |                        |
  |------- FIN(seq=u) ---->|  第一次挥手：客户端说"我发完了"
  |                        |
  |<------ ACK(ack=u+1) ---|  第二次挥手：服务端说"知道了"
  |                        |
  |   (服务端可能还有数据要发)  |
  |                        |
  |<------ FIN(seq=w) -----|  第三次挥手：服务端说"我也发完了"
  |                        |
  |------- ACK(ack=w+1) -->|  第四次挥手：客户端说"收到"
  |                        |
  |   TIME_WAIT (2MSL)     |
  |                        |
```

为什么不能三次挥手？因为服务端收到客户端的 FIN 时，可能还有数据没发完。它先 ACK 表示"知道你要关了"，等数据发完，再发 FIN。

当然如果服务端恰好没数据要发了，第二次和第三次可以合并，实际抓包有时候能看到三次挥手的情况。

## TIME_WAIT 是什么

主动关闭方在四次挥手后不是直接关闭，而是进入 TIME_WAIT，等 2 个 MSL（报文最大存活时间，通常 60 秒）才真正关闭。

两个原因：

1. **保证最后的 ACK 到达**：如果最后一个 ACK 丢了，服务端会重发 FIN。处于 TIME_WAIT 的客户端还能收到并重发 ACK。如果客户端已经关了，服务端重发的 FIN 没人理，就卡在 LAST_ACK 了。

2. **让旧报文消亡**：等 2MSL 确保网络中残留的报文都过期了，不会干扰后面用相同端口的新连接。

TIME_WAIT 在高并发短连接场景下是个问题，大量连接卡在这个状态占端口。可以调内核参数 `tcp_tw_reuse` 来优化。

我之前在服务器上跑 `netstat` 看到过几万个 TIME_WAIT，当时还以为是连接泄露，查了半天才知道是正常的。

## 几个常见面试题

**SYN 洪泛攻击是什么？**

攻击者发大量 SYN 包但不回 ACK（第三次握手），服务端维护大量半连接（SYN_RCVD 状态），耗尽资源。防御方式：SYN Cookie，让服务端不保存半连接状态。

**三次握手可以携带数据吗？**

前两次不行，第三次可以。因为第三次握手的时候客户端已经确认连接建立了，带数据是安全的。

**序列号为什么是随机的？**

安全考虑。如果序列号是固定的或者容易预测，攻击者可以伪造 TCP 包。随机 ISN 让伪造变困难。

说到这里，TCP 的连接管理看着简单，实际设计考虑了很多边界情况。建议自己用 Wireshark 抓个包看看，会发现理论和实际还是有些区别的。比如 TCP Fast Open 可以在 SYN 包里就带数据，减少一个 RTT。不过这些进阶内容一般面试不会问太深。
