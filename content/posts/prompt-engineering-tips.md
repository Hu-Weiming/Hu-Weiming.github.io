# Prompt Engineering 实用技巧，我总结了这些

用大模型一年多了，写了不少 Prompt。从最开始的"帮我写个 xxx"，到现在能让模型按我要的格式、风格来输出，踩了不少坑。总结一下真正有用的技巧。

## 零样本和少样本提示

**零样本（Zero-shot）**：直接给指令，不给例子。

```
请将以下英文翻译成中文：Hello, how are you?
```

简单任务够了。但复杂点的任务，模型可能猜不准你要什么。

**少样本（Few-shot）**：给几个示例。

```
请对以下评论进行情感分类。

评论：这家店太好吃了，下次还来！
分类：正面

评论：等了一个小时才上菜，再也不来了
分类：负面

评论：东西还行，但价格偏贵
分类：
```

给了示例，模型就知道格式了，输出稳定很多。

我的经验：能用少样本就用少样本。虽然多费 Token，但质量提升明显。

## 角色设定

System Prompt 里给模型一个角色：

```
你是一个有10年经验的 Java 后端工程师，擅长 Spring Boot 和微服务架构。
回答简洁专业。
```

回答会更有针对性。我审查代码的时候经常用："你是一个代码审查专家，找出 bug 和可以优化的地方。"效果比"看看这代码有没有问题"好太多。

## 思维链（Chain of Thought）

这是个很强的技巧。让模型一步步推理，而不是直接给答案。

最简单的方式，加一句"请一步步思考"：

```
一个水池有两个进水管和一个出水管。
进水管A每小时进3吨水，进水管B每小时进5吨水。
出水管每小时出2吨水。水池容量20吨，多久能装满？

请一步步思考。
```

加了"一步步思考"之后，模型会把推理过程写出来，准确率高很多。不加的话，复杂一点的推理题容易直接给错答案。

更进阶的是 Few-shot CoT，在示例中就展示推理过程：

```
问题：小明有5个苹果，吃了2个，又买了3个，现在有几个？
思考：小明开始有5个，吃了2个剩3个，又买了3个，3+3=6个。
答案：6个

问题：...（你的实际问题）
```

我做算法题的时候经常让模型先分析思路再写代码，比直接让它写代码靠谱多了。

## 结构化输出

如果你需要程序处理模型的输出，结构化很重要。直接告诉模型你要什么格式：

```
请分析以下文本的情感，以 JSON 格式返回：
{
  "sentiment": "正面/负面/中性",
  "confidence": 0.0-1.0,
  "keywords": ["关键词1", "关键词2"]
}

文本：今天天气真好，心情也不错，就是午饭有点难吃。
```

模型通常能很好地遵循 JSON 格式。如果还不够稳定，可以在 few-shot 里给个完整示例。

OpenAI 和一些模型还支持 JSON Mode，强制输出合法 JSON。LangChain4j 里也有结构化输出的支持，直接映射成 Java 对象。
