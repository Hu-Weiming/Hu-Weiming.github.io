# Prompt Engineering 实用技巧，我总结了这些

用大模型一年多了，写了不少 Prompt。从最开始"帮我写个 xxx"的大白话，到现在能让模型按我要的格式和风格输出，踩了不少坑。总结一下真正实用的技巧。

## 零样本和少样本提示

**零样本（Zero-shot）**：直接给指令。

```
请将以下英文翻译成中文：Hello, how are you?
```

简单任务零样本就够了。

**少样本（Few-shot）**：给几个示例让模型"学"格式。

```
请对以下评论进行情感分类。

评论：这家店太好吃了，下次还来！
分类：正面

评论：等了一个小时才上菜，再也不来了
分类：负面

评论：东西还行，但价格偏贵
分类：
```

给了示例，输出格式稳定很多。我的经验：能用少样本就用少样本，虽然多费 Token，但质量提升明显。

## 角色设定

System Prompt 里给角色：

```
你是一个有10年经验的 Java 后端工程师，擅长 Spring Boot 和微服务架构。
回答简洁专业，附带代码示例。
```

回答会更有深度和针对性。我审查代码经常用"你是代码审查专家"这种角色。效果比"看看这代码有没有问题"好太多。

角色设定还有个好处：限定模型的回答范围。比如你设定"你是一个 Java 专家"，它就不太会扯到 Python 的方案去。

## 思维链（Chain of Thought）

让模型一步步推理，而不是直接给答案。最简单的方式——加一句"请一步步思考"：

```
一个水池有两个进水管和一个出水管。
进水管A每小时3吨，进水管B每小时5吨。
出水管每小时出2吨。水池容量20吨，多久能装满？

请一步步思考。
```

模型会把推理过程写出来，准确率高很多。不加的话复杂推理题容易直接算错。

更进阶的是 Few-shot CoT——在示例中就展示推理过程：

```
问题：小明有5个苹果，吃了2个，又买了3个，现在几个？
思考：开始5个，吃了2个剩3个，买了3个，3+3=6。
答案：6个

问题：（你的实际问题）
```

我做算法题的时候经常让模型先分析思路再写代码，比直接让它写靠谱很多。

## 结构化输出

需要程序处理模型输出时，结构化很关键：

```
请分析以下文本的情感，以 JSON 格式返回：
{
  "sentiment": "正面/负面/中性",
  "confidence": 0.0-1.0,
  "keywords": ["关键词1", "关键词2"]
}

文本：今天天气真好，心情也不错，就是午饭有点难吃。
```

模型通常能很好地遵循 JSON 格式。OpenAI 还支持 JSON Mode 强制输出合法 JSON。

在 LangChain4j 里可以直接定义 Java 对象来接收：

```java
record SentimentResult(String sentiment, double confidence, List<String> keywords) {}

interface Analyzer {
    @UserMessage("分析以下文本的情感：{{text}}")
    SentimentResult analyze(@V("text") String text);
}
```

框架会自动处理 JSON 解析，很方便。

## 实际使用中的小技巧

分享几个我日常用大模型的技巧：

**1. 给限制条件**

"用 Java 17 语法"、"不要使用第三方库"、"代码不超过50行"。限制越明确，输出越符合预期。

**2. 让模型问你问题**

如果你的需求不太明确，可以说："在开始之前，先问我几个问题来确认需求。"模型会主动澄清不确定的点。

**3. 分步骤给任务**

复杂任务别一股脑全丢给模型。拆成几步：先让它设计方案，确认后再让它实现，最后让它审查。

**4. 否定句不如肯定句**

"不要用递归"不如"请用迭代方式实现"。模型对肯定指令的遵循度更高。

**5. 重要的事说三遍（不是）**

重要的约束可以在 Prompt 里强调，放在开头和结尾各说一次，模型会更重视。

这些技巧看着简单，但真的好用。我之前写 Prompt 总觉得模型"不听话"，后来发现多半是我表达不够清楚。Prompt Engineering 说到底就是跟模型沟通的艺术，把你的需求表达得越精确，输出就越好。
